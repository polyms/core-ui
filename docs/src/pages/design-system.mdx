**Polyms Core UI** (`@polyms/core-ui`) is a modular React component library shipped from this monorepo: **React 19**, **TypeScript (strict)**, **Tailwind CSS 4**, primitives from **@base-ui/react**, and **Floating UI** where overlays need positioning. This page summarizes the **visual and interaction contracts** consumers and contributors should follow; the npm package also ships **`core/AI.md`** with the same consumption rules—if anything disagrees with **`index.d.ts`** for your installed version, trust the types.

## Philosophy

The system is built on three principles:

**Consistency** — One semantic palette (primary, success, warning, danger, informational accents, light/dark surfaces) plus a **slate** gray ramp for chrome, so composed UI stays visually aligned with bundled components.

**Accessibility** — Semantic HTML, keyboard support, and ARIA patterns from **Base UI** primitives; target **WCAG 2.1 AA** for text contrast and focus behavior in product screens.

**Flexibility** — Themeable **CSS custom properties** (colors, z-index scale) and Tailwind utilities; prefer **component props** and **documented composition trees** over re-styling library-owned shells.

## Foundation

### Color System

Semantic ramps use **11 steps (50–950)** where the theme defines them. Prefer **`var(--color-{name}-{step})`** in CSS and **`bg-*` / `text-*` / `border-*`** in Tailwind for **primary**, **success**, **info**, **warning**, and **danger**. For **neutral chrome** (borders, muted text, surfaces next to library components), use **slate only**—not the Tailwind **`neutral`** scale (see below).

```css
/* CSS Custom Properties */
var(--color-primary-500)
var(--color-success-600)
var(--color-danger-50)

/* Tailwind Utilities */
bg-primary-500
text-success-600
border-danger-50
```

**Tailwind color utilities**

- `bg-{semantic}-{50…950}` — background
- `text-{semantic}-{50…950}` — foreground
- `border-{semantic}-{50…950}` — borders

**Do not** substitute unrelated Tailwind families (`emerald-*`, `rose-*`, `amber-*`, …) when you mean **success**, **danger**, or **warning**—that drifts away from the contract used by **`btn-*`**, **`badge-*`**, and **`alert-*`**.

#### Slate vs neutral

Apps that compose this library should use **`text-slate-*`**, **`bg-slate-*`**, **`border-slate-*`**, and **`var(--color-slate-50)` … `var(--color-slate-950)`** for generic gray UI. Avoid **`neutral-*`** utilities and **`var(--color-neutral-*)`** next to core-ui components so borders, text, and surfaces stay on the same hue and contrast curve.

### Spacing System

Spacing follows **Tailwind CSS 4**’s default scale (typically **0.25rem** steps via `1` = `0.25rem`, etc.).

```tsx
<div className='p-4 gap-2 mb-6'>
  {/* padding: 1rem, gap: 0.5rem, margin-bottom: 1.5rem */}
</div>
```

For **RTL-friendly** layouts, prefer **logical** utilities — `ms-*` / `me-*`, `ps-*` / `pe-*`, `border-s-*` / `border-e-*` — over physical `ml` / `mr` / `pl` / `pr` in new code (see **`core/AI.md`** in the package).

## CSS-only patterns and item-*

Several patterns are **CSS classes only** (no dedicated React export). They are part of the public surface once **`@polyms/core-ui`** styles are loaded:

- **Tables:** `.table`, `.table-sm`, `.table-lg`, `.table-striped`, `.table-hover`, `.table-bordered`, `.table-borderless`, `.table-active`, `.table-group-divider`, `.caption-top`, **`thead.thead-light`**
- **Badge / card / link:** `.badge` (+ size and tone variants), `.card`, `.card-body`, `.link` + `.link-primary` / `.link-danger` / `.link-light`, `.stretched-link`
- **Chrome helpers:** `.border-light`, `.border-{t|r|b|l|s|e|x|y}-light`, `.bg-light`
- **Skeleton / type scale:** `.skeleton`, `.h1` … `.h6`
- **Layout:** `.@page`, `.container-page` (container-query page layout)

**`item-*`** classes implement the shared **ghost** interaction recipe (transparent idle → tinted hover/active) for nav rows, chips, list rows, and ad-hoc controls. Variants: **`item-primary`**, **`item-success`**, **`item-info`**, **`item-warning`**, **`item-danger`**, **`item-light`**, **`item-dark`**. Combine with **`.active`** for a persistent selected state (same look as `:active`). Prefer these over one-off `hover:bg-*` chains so tones stay aligned with buttons and menus.

Full tables, examples, and AI-generation rules live in **`core/AI.md`** (shipped in the npm package).

## Component Architecture

### Stack and exports

- **Single barrel import:** `import { Button, Modal, Field, … } from '@polyms/core-ui'` — avoid deep paths unless your toolchain explicitly supports them.
- **Peers:** `react` and `react-dom` **>= 19**.
- **Primitives:** Many building blocks re-export **@base-ui/react** (e.g. `Accordion`, `Toggle`, `ToggleGroup`); internal components use **`useRender`** from **`@base-ui/react/use-render`** for polymorphic `render` props.
- **Styles:** Import the built **`styles-*.css`** from the package root, or the **Tailwind 4** entry **`styles/tailwind.css`** when compiling tokens in your app pipeline.

### Composition pattern

Components use **compound (dot) notation** — keep **Trigger → Content → semantic slots** (`Header`, `Body`, `Footer`, `Item`, …) in the documented order so accessibility and bundled CSS selectors stay valid.

```tsx
<Alert variant='success' onDismiss={handleDismiss}>
  <Alert.Heading>Success!</Alert.Heading>
  Payment processed successfully.
</Alert>

<Modal>
  <Modal.Trigger>Open</Modal.Trigger>
  <Modal.Content size='lg' title='Account settings'>
    <Modal.Header>Account settings</Modal.Header>
    <Modal.Body>{/* primary content */}</Modal.Body>
    <Modal.Footer>
      <Button variant='primary'>Save</Button>
    </Modal.Footer>
  </Modal.Content>
</Modal>

<Field name='email'>
  <Field.Label>Email</Field.Label>
  <Field.Control />
  <Field.Description>We never share your email.</Field.Description>
</Field>

<BottomSheet>
  <BottomSheet.Trigger>Open sheet</BottomSheet.Trigger>
  <BottomSheet.Content title='Actions'>{/* mobile-first panel */}</BottomSheet.Content>
</BottomSheet>
```

### Built-in surfaces

Roots like **`Modal.Content`**, **`Offcanvas.Content`**, **`Menu.Content`**, and drawer internals already encode **padding, layout, motion, and z-index**. Prefer **props** (`size`, `centered`, `scrollable`, `variant`, …) and **slot components** over pasting large Tailwind bundles onto those nodes.

### CSS variables pattern

Components use CSS custom properties for themeable styling:

```css
.alert {
  background-color: var(--alert-bg);
  border-color: var(--alert-border);
  color: var(--alert-color);
}

.alert-success {
  --alert-bg: var(--color-success-50);
  --alert-border: var(--color-success-600);
  --alert-color: var(--color-success-700);
}
```

### Data attributes

Base UI and Polyms wrappers expose **`data-*`** hooks for state and placement. Prefer them in **component CSS** rather than ad-hoc global selectors:

```css
[data-part='trigger'][data-state='open'] {
  /* open trigger */
}

[data-part='item'][data-highlighted] {
  /* highlighted collection item */
}

[data-part='content'][data-placement='top'] {
  /* placement-specific content */
}
```

### Toasts and imperative modals

- **Toast:** wrap the app with **`Toast`** and render **`Toast.Container`** where toasts should portal.
- **Programmatic modals:** **`useModalStore`** (**`showModal`** / **`closeModal`**) requires **`zustand`** in the consumer app and a mounted **`Modal.Container`** when using the store API.

## Usage Guidelines

### Color usage

**Do:**

- Use **semantic** ramps for intent (success / warning / danger / primary / info).
- Use **slate** for neutral chrome; keep **4.5:1** contrast for body text where feasible.
- Use lighter steps (50–200) for backgrounds and darker steps (600–900) for text and borders on light surfaces.

**Don’t:**

- Mix meanings (e.g. success styling on destructive actions).
- Pair **slate-based** library UI with the **`neutral`** Tailwind scale in the same chrome without a deliberate exception.
- Override semantic tokens with arbitrary unrelated palettes for the same role.

### Component selection

Choose by **interaction and attention**:

- **Modal** — Blocking, focused work; centered dialog pattern with **`Modal.Content`** sizing and header/body/footer slots.
- **Offcanvas** — Navigation and supplementary panels that slide from an edge.
- **BottomSheet** — **Mobile-first** sheet (title bar, optional back/close); theme uses the same **modal-tier** z-index variables as dialogs so layers stay consistent.
- **Popover** — Contextual, non-blocking content.
- **Tooltip** — Short hints on hover/focus.

### Accessibility

- Prefer **library slots** for titles and descriptions so **`aria-*`** relationships stay wired.
- Preserve **keyboard** roving focus for menus, toolbars, and composite widgets.
- Manage **focus** when opening/closing dialogs and drawers per component docs.
